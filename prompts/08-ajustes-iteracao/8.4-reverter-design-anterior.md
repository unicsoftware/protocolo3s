# 8.4 — Reverter pro design anterior (quando a IA piorou)

> **Tópico:** Ajustes & Iteração
> **Objetivo:** Quando uma iteração com a IA estragou o que estava bom, voltar à versão anterior **sem perder o que era válido**.

## Quando usar

Você pediu uma alteração simples. A IA reescreveu meia página. O cliente preferia a versão anterior. Mas vocês já perderam o código original (não salvou). Este prompt **reconstrói** a versão "boa" a partir do que você lembra dela.

## O Prompt

```
Você é um designer-investigador. A IA pisou no calo: alteração feita destruiu mais do que melhorou. Sua tarefa é RECONSTRUIR a versão anterior (a que estava boa) a partir do código atual + descrição do que era melhor antes.

CÓDIGO ATUAL (pós-piora):
{{CODIGO_PIORADO}}

DESCRIÇÃO DO QUE ERA MELHOR ANTES:
{{DESCRICAO_ANTERIOR}}
(ex: "tinha 3 cards menores em vez de 1 grande", "o título era em serifa, agora ficou sans", "tinha foto à esquerda e texto à direita, agora empilhou")

O QUE A IA FEZ DE BOM (e dá pra preservar):
{{ASPECTOS_BONS_NOVOS}}
(ex: "a copy nova é melhor que a antiga", "a cor do botão ficou ok")

REGRAS:
1. RECONSTRUA estrutura, layout e hierarquia que o usuário descreveu como "melhor antes".
2. PRESERVE elementos novos válidos (copy nova, cor melhor, ícone melhor) — só se forem realmente melhores e couberem.
3. Se houver dúvida sobre algum detalhe (ex: tamanho exato, espaçamento), faça suposição razoável e MARQUE com [SUPOSIÇÃO — confirmar].
4. Não "tente fazer um meio termo" — se a versão antiga era melhor, é ela. O meio termo costuma ser pior que ambas.
5. Se a descrição é vaga ("tava melhor"), liste 3-5 perguntas específicas antes de reconstruir.

ENTREGA:
1. Código reconstruído da versão "boa", com correções herdadas da nova quando válidas.
2. Lista do QUE VOLTOU AO ANTERIOR (estrutura, layout, hierarquia).
3. Lista do QUE FOI PRESERVADO DA VERSÃO NOVA (apenas aspectos bons).
4. Lista de SUPOSIÇÕES feitas, marcadas claramente.
5. Sugestão prática: "para evitar perder versões boas no futuro, use git/snapshots — segue exemplo no fim".

Agora execute.
```

## Variáveis

| Variável | O que preencher |
|---|---|
| `{{CODIGO_PIORADO}}` | Versão atual (a ruim) |
| `{{DESCRICAO_ANTERIOR}}` | O que tinha de bom antes — seja específico |
| `{{ASPECTOS_BONS_NOVOS}}` | O que dá pra manter da nova versão |

## Dicas de uso

- **Salva snapshot a cada milestone.** No Lovable usa o histórico interno. Em projeto local, comita no git antes de cada chamada grande de prompt-mestre.
- Se você **NÃO consegue descrever** o que era melhor, peça ao cliente: "me diz 3 coisas específicas que mudaram pra pior". A resposta dele vira o input.
- Quando o cliente diz "tá pior", às vezes ele só **não está acostumado**. Espera 24h, mostra de novo. Metade das vezes ele aprova.
- A **sugestão de git/snapshots** no final do prompt vira hábito — adiciona o lembrete sempre.
