# 2.3 — Uma página vs multi-página

> **Tópico:** Estrutura & Arquitetura
> **Objetivo:** Decisão binária com critério: o site deste cliente deve ser **single-page** (rola tudo) ou **multi-página** (com menu navegável)?

---

## Quando usar

Logo após o briefing. Decisão errada custa caro: single-page para um advogado com 8 áreas distintas vira muralha; multi-página para uma barbearia com 4 serviços vira labirinto vazio.

## Como usar

1. Preencha 6 variáveis curtas.
2. Saída: recomendação binária + justificativa + estrutura sugerida (lista de páginas se for multi-página, ou ordem de seções se for single).

## O Prompt

```
Você é um arquiteto de informação. Sua tarefa é decidir entre site de UMA PÁGINA (long-scroll) ou MULTI-PÁGINA (com menu navegável) para um cliente específico.

DADOS DE ENTRADA:
- Nicho: {{NICHO}}
- Quantidade de serviços / produtos distintos: {{QTD_SERVICOS}}
- Existe necessidade de conteúdo profundo por serviço? (ex: explicação técnica, FAQ específico): {{CONTEUDO_PROFUNDO}}
- Volume de fotos / portfolio: {{VOLUME_FOTOS}}
- Necessidade de blog / SEO de conteúdo: {{BLOG_SEO}}
- Tráfego predominante: {{TRAFEGO}}
- Maturidade digital do cliente (vai conseguir manter o site?): {{MATURIDADE}}

SUA TAREFA:
Decidir entre single-page e multi-página, com base em critérios objetivos. Sem "depende, vai do gosto".

CRITÉRIOS DE DECISÃO (use estes — não invente outros):
1. SINGLE-PAGE quando: serviços ≤ 6, cada serviço cabe em 2-3 linhas, sem necessidade de SEO de conteúdo, tráfego vem de Insta/anúncio/indicação.
2. MULTI-PÁGINA quando: serviços ≥ 7 OU cada serviço pede explicação longa, OU há blog, OU SEO orgânico é estratégico, OU portfolio extenso (>30 fotos).
3. Em caso de empate, escolha SINGLE-PAGE — é mais simples de manter, mais rápido de carregar, mais barato de gerar.
4. Se a maturidade do cliente for BAIXA, evite multi-página: ele não vai atualizar.

FORMATO DE SAÍDA:

## Decisão
**[SINGLE-PAGE | MULTI-PÁGINA]**

## Justificativa
[3-5 linhas, citando os critérios que pesaram]

## Estrutura recomendada

(se SINGLE-PAGE)
Liste as seções em ordem vertical, do topo ao rodapé.

(se MULTI-PÁGINA)
Liste as páginas e, dentro de cada página, as seções principais.

## Atenção / risco
[1-2 pontos: o que pode dar errado com essa escolha e como mitigar]

## Quando reavaliar
[gatilho que justifica migrar para o outro modelo no futuro — ex: "quando atingir 30+ posts de blog", "quando lançar o 7º serviço"]

Agora execute.
```

## Variáveis

| Variável | O que preencher |
|---|---|
| `{{NICHO}}` | Tipo de negócio |
| `{{QTD_SERVICOS}}` | Número aproximado de serviços/produtos distintos |
| `{{CONTEUDO_PROFUNDO}}` | "sim" ou "não" — cada serviço pede explicação longa? |
| `{{VOLUME_FOTOS}}` | "poucas (<10)", "médio (10-30)", "grande (>30)" |
| `{{BLOG_SEO}}` | "sim, é estratégico" / "não, não vai postar" |
| `{{TRAFEGO}}` | Insta / anúncio / Google orgânico / indicação |
| `{{MATURIDADE}}` | Baixa / média / alta |

## Dicas de uso

- **80% dos casos do protocolo 3s = single-page.** Multi-página só entra quando há volume real.
- Se o cliente "quer ter blog mas nunca vai postar" — single-page. Não construa o que vai apodrecer.
- Output deste prompt é **input direto do 2.1** — a decisão muda quais seções vão entrar.
- Cliente romântico que pede menu enorme com "Sobre", "Equipe", "Histórico", "Missão", "Visão", "Valores" — explica que tudo isso cabe em 1 seção bem feita. Use a saída deste prompt como argumento.
